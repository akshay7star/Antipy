# Production Logging

When Junior engineers code scripts on their laptops, they utilize `print()` to output text to their Terminal. 

In a Production Cloud Server (like AWS or Google Cloud), there is no screen. Nobody is sitting at a monitor exclusively watching your Terminal window. If your algorithm crashes organically at 3:00 AM and relies on a `print("Database disconnected")` statement, that data pointer violently disappears into the abyss forever.

Enterprise software systems categorically ban `print()`. Instead, they exclusively deploy Python's built-in **`logging`** engine to rigorously document application state securely into permanent File Descriptor pipelines.

## 1. The Logging Severity Matrix

The `logging` module mathematically grades system events strictly based on a standardized 5-Tier Severity Matrix. This allows DevOps engineers to filter out standard "Okay" messages and trigger PagerDuty text messages specifically exclusively on "Critical" crashes.

1. **DEBUG (10)**: Verbose data matrices. Used specifically when actively hunting down logic bugs.
2. **INFO (20)**: Normal operational analytics (e.g., "User logged in natively").
3. **WARNING (30)**: The application successfully executes, but strictly detects an anomaly (e.g., "Slow network response").
4. **ERROR (40)**: A function catastrophically crashed, but the overall server remains actively alive natively.
5. **CRITICAL (50)**: Absolute system meltdown. A core database failed.

# 1. INITIALIZING THE ENGINE
# The basicConfig natively explicitly binds the engine to a physical log file mathematically!
# It also defines a strictly standardized Timestamp string formatting structural template.

    filename='production_server.log',
    level=logging.DEBUG, # Captures EVERYTHING ranked Level 10 or mathematically higher
    format='%(asctime)s | [%(levelname)s] | %(message)s'
)

# 2. DISPATCHING MESSAGES

## 2. Handlers and Architecture

A `logger` object does not just write to a single destination. 

Advanced engineering relies on **Handlers**. You can configure a single `logger.error("Crash")` command to natively physically execute multiple disparate hardware mechanisms simultaneously.

- **StreamHandler**: Triggers standard terminal output for local developers.
- **FileHandler**: Appends the text securely string directly onto a local `.log` file payload.
- **HTTPHandler**: Beams the exact error text packet directly over the internet securely into a DataDog or Splunk analytic dashboard.

<interactive-code>
import logging
import sys

# 1. GENERATE A CUSTOM LOGGER MATRIX
logger = logging.getLogger("EnterpriseSystem")
logger.setLevel(logging.INFO)

# 2. INSTANTIATE DISTINCT HANDLERS
# Handler A: Outputs dynamically natively to the Terminal Window dynamically.
console_engine = logging.StreamHandler(sys.stdout)

# Handler B: Pipes structurally exclusively directly mathematically into a File.
file_engine = logging.FileHandler('secure_audit.log')

# 3. APPLY DISTINCT FORMATTERS
formatter = logging.Formatter('[%(name)s] %(levelname)s - %(message)s')
console_engine.setFormatter(formatter)
file_engine.setFormatter(formatter)

# 4. BIND ENGINES TO LOGGER
logger.addHandler(console_engine)
logger.addHandler(file_engine)

logger.info("This exact single string is physically transmitted seamlessly identically across both independent Hardware I/O pipes flawlessly!")
</interactive-code>
